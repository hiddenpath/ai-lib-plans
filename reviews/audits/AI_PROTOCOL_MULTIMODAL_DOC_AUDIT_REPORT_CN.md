## 1. 执行摘要

两份文档总体质量较高，具备明显的工程组织价值，但存在“方法学不完整、与协议驱动约束存在偏差、跨运行时一致性门禁不足”三类核心问题。  
结论为：

- **可采纳**：整体方向与任务拆分框架
- **需先修正**：事实/推测混用、schema 对齐不充分、compliance-first 门禁缺失
- **建议策略**：先文档纠偏和验收口径统一，再推进实现

---

## 2. 合理性评估（优点）

## 2.1 调研报告（survey）优点

- 结构完整，包含执行摘要、provider 分析、映射可行性、风险与路线图
- 多 provider 评估维度相对统一，便于横向比较
- 对非官方生态（Suno/Runway/Midjourney）给出了风险意识与谨慎结论
- 输出了 P0/P1/P2 优先级，具备转任务基础

## 2.2 集成计划（integration）优点

- 任务颗粒度清晰，包含依赖、工时、验收点的雏形
- 覆盖协议、运行时、测试、文档等环节，具备执行路径
- 明确了跨语言实施意识（Rust/Python/TS）

---

## 3. 不足与问题清单

## 3.1 P0 级问题（必须先修）

1. **事实与推测未严格分离**  
   - 存在“推测 API 结构”与“已验证事实”混写，影响结论可信度与可复现性。

2. **与 ARCH-001 存在偏离风险**  
   - 文档中有“按 provider 特化 runtime 逻辑”的倾向，可能演变为 runtime 硬编码，违背“协议驱动、配置优先”的原则。

3. **与 ARCH-003 对齐不足**  
   - 缺少明确的跨运行时一致性门禁（同输入同语义输出），`ai-protocol/tests/compliance/` 未被设为主验收。

4. **schema 与现状差距未显式管理**  
   - 一些新增字段建议没有清楚标注“现有支持/需 schema 扩展/仅实验约定”，后续容易产生兼容性争议。

## 3.2 P1 级问题（短期修复）

1. **任务依赖与范围前后不一致**  
   - 个别任务依赖关系与目标 provider 或能力不一致，增加排期风险。

2. **验收标准缺少量化阈值**  
   - 延迟、错误率、内存、回滚触发阈值未量化，QA 难以客观判定。

3. **回滚策略不完整**  
   - 缺少 feature flag、灰度策略、触发阈值、执行路径。

## 3.3 P2 级问题（中长期优化）

1. 文档版本漂移治理不足（价格/rate-limit/模型能力随时间变化）
2. 跨语言运行时差异清单缺失（Rust/Python/TS）
3. 缺乏可复用“最小参考实现+失败注入”附录模板

## 4. 必要补充与调整（可执行）

## 4.1 P0（本周内）

- 新增“评估方法与证据等级”章节  
  - 评分维度、权重、证据等级（A 官方、B 实测、C 推测）
- 强制“事实/推测分栏”  
  - 所有推测项进入独立章节，并标注 `UNVERIFIED`
- 新增“schema 对齐差距表”  
  - 字段级标注：`supported` / `needs_schema_change` / `experimental`
- 增加 compliance-first 门禁  
  - 文档中明确：实现验收必须包含 `ai-protocol/tests/compliance/`

## 4.2 P1（1-2 周）

- 追加“跨运行时验证矩阵”  
  - Provider x 模态 x 传输模式 x 错误场景
- 增加回滚与灰度章节  
  - 含触发阈值与执行流程
- 统一任务 DoD 模板  
  - 每项任务必须有：功能验收 + 测试验收 + 文档验收

## 4.3 P2（持续）

- 增加文档“最后验证日期/来源链接”治理机制
- 建立跨运行时差异跟踪页
- 建立最小可运行样例库和失败注入样例

---

## 5. 风险与回滚建议

## 5.1 风险

- 文档先行但约束不清，导致实现反复返工
- 跨运行时语义漂移，导致上层应用行为不一致
- 未定义回滚阈值，线上问题响应慢

## 5.2 回滚建议

- 新增能力默认受 feature flag 控制
- 任何新增字段需声明兼容策略（可选、默认空、非破坏）
- 里程碑验收前执行一次“回滚演练”

---

## 6. 验收标准（本轮文档纠偏）

1. 两份原文档完成修订提纲对齐（方法、证据、门禁、回滚）
2. 形成跨运行时验证矩阵初稿
3. 新增任务卡（PT）覆盖 P0/P1 落地项
4. `MEMORY.md` 写入长期约束条目

1. 文档纠偏（P0）
2. 任务重排与验收门禁统一（P1）
