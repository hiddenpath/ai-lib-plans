# spiderswitch 运行时路由能力评估报告（重写版）

**报告日期**: 2026-03-06  
**范围**: `spiderswitch`、`ai-lib-rust`、`ai-lib-ts`（对比维度）  
**定位**: 架构与策略评估（内部）

---

## 1. 目标与约束

基于当前共识，`spiderswitch` 的职责边界明确为：

1. **提供底层运行时路由能力**（Runtime Routing Capability）
2. **不内置复杂策略引擎**，策略由上层应用（Agent/Orchestrator）制定与驱动
3. 路由目标不只限于当前三种运行时，需支持未来 `go`、`wasm` 等扩展
4. 路由主需求是**大模型能力分发**，同时必须考虑运行时自身能力（稳定性、延迟、工具链、可观测性）

---

## 2. 执行摘要（结论）

### 2.1 是否需要动态运行时路由

**需要**。必要性来自四类现实场景：

- 能力分发：按 `tools/vision/audio/streaming` 等能力把请求送到合适 runtime/provider 组合
- 可用性保障：在区域网络、凭证、限流、异常分类差异下做可控降级
- 演进兼容：支撑 Python -> Rust/TS 的渐进迁移与灰度
- 多目标优化：上层按成本、时延、稳定性策略选择最优运行时

### 2.2 spiderswitch 应做什么，不应做什么

**应做**：
- 统一 runtime 抽象与注册机制
- 暴露可路由的能力与状态信号
- 执行“被动路由”（接收上层决策并落地）

**不应做**：
- 内置复杂策略决策（如权重学习、业务优先级、租户级预算分配）
- 在控制平面里耦合业务规则

### 2.3 是否应移植到 Rust 或 TS

**结论**：迁移会提升运行时路由承载能力，但不是“移植即自动变好”。  
真正提升来自“统一抽象 + 能力模型 + 可观测性 + 契约测试”。

- **Rust**：更适合高并发、低延迟、强可靠路由控制平面
- **TS**：更适合 Web/Edge 场景、前后端同构和快速集成
- **Python**：仍适合快速迭代与生态对接，不应直接判定为淘汰

---

## 3. 现状判断（spiderswitch Python 版）

当前 `spiderswitch` 已有 `Runtime` 抽象和工具路由入口，具备演进基础；但整体仍是“单运行时实例优先”的进程模型。  
这意味着它可以支持“切换模型”，但尚未完整支持“多运行时并行路由编排”。

### 3.1 现有优势

- 已有 runtime 抽象、状态管理、工具分发与错误封装
- 能力与就绪信号（如 API key / proxy 状态）已进入工具输出
- 对并发切换有基础保护（避免同 runtime 下并发竞态）

### 3.2 现有限制

- 单实例 runtime 绑定较强，缺少统一 runtime registry/resolver
- 进程级环境变量副作用对多 runtime 隔离不友好
- 状态模型尚未天然支持“多 runtime 维度”的并行观测

---

## 4. Rust / TS 对运行时路由支持的对比评估

## 4.1 Rust 方向

**优势**：
- 高并发与资源控制更强，适合作为路由控制平面核心
- 错误分类、重试/熔断/限流等弹性策略组件成熟度高
- 类型系统有利于维护跨 runtime 的严格契约

**风险/成本**：
- 初期开发和调试门槛较高
- MCP server 生态与工具链集成需要补齐工程脚手架

**适配建议**：
- 把 Rust 作为高可靠控制平面的首选实现
- 优先构建能力路由基础设施，再逐步替换执行后端

## 4.2 TS 方向

**优势**：
- 对前端、Node、Edge、Serverless 生态接入成本低
- JSON/MCP 交互开发速度快，易与上层编排系统联动
- 对未来 WebAssembly/跨端编排的桥接潜力较好

**风险/成本**：
- 极限性能与稳定性上限通常弱于 Rust
- 长时间高并发场景需严格治理内存和事件循环压力

**适配建议**：
- 作为集成层、边缘层、控制台层实现更有优势
- 可与 Rust 形成“TS 编排 + Rust 核心执行”的组合

## 4.3 Python 方向（继续保留价值）

**保留理由**：
- 生态成熟，原型验证与问题定位效率高
- 对历史行为兼容性最好，适合作为迁移期基准实现

**定位建议**：
- 保持为参考实现和回退路径
- 将策略逻辑外置，减轻 Python 版控制平面复杂度

## 5. 未来多运行时扩展（含 Go / WASM）

为避免未来每新增一个 runtime 都进行大改，建议采用 runtime-neutral 架构。

### 5.1 统一运行时能力模型（必须）

为每个 runtime 声明标准化能力与约束字段（示例）：

- `runtime_id`: `python` / `rust` / `ts` / `go` / `wasm`
- `capabilities`: `chat`, `streaming`, `tools`, `vision`, `audio`, `embeddings`
- `operational`: `max_concurrency`, `cold_start_ms`, `supports_proxy`, `supports_hot_reload`
- `reliability`: `retry_classes`, `circuit_breaker`, `rate_limit_adaptive`

### 5.2 Runtime Adapter 契约（必须）

定义统一 SPI（接口语义），至少包括：

- `list_models(filters)`
- `switch_model(target)`
- `health/runtime_status()`
- `close()/reset()`

并约束：

- 错误类型统一映射
- 状态与 epoch 语义一致
- 能力声明可被上层策略直接消费

### 5.3 路由模式（建议）

- **Mode A: 手动指定 runtime**（运维/调试）
- **Mode B: 上层策略下发 runtime 选择结果**（推荐主模式）
- **Mode C: 可选默认回退链**（仅做安全兜底，不做业务策略）

## 6. 运行时路由需求场景与利弊

### 场景 1：按模型能力分发（主场景）

**触发**：请求声明需要 `tools` 或 `vision` 等能力。  
**收益**：减少能力不匹配失败，提升首包成功率。  
**代价**：依赖能力元数据准确性，需要持续校验。

### 场景 2：按运行时能力分发

**触发**：高并发/低延迟任务优先 Rust，快速实验任务优先 Python/TS。  
**收益**：把“模型能力”和“运行时能力”同时纳入决策。  
**代价**：调度维度增加，观测与调试复杂度上升。

### 场景 3：按地域与网络条件分发

**触发**：代理可用性、出口策略、区域限流差异。  
**收益**：可用性提升，减少不可达导致的失败。  
**代价**：网络信号存在噪声，需要结合真实错误分类。

### 场景 4：迁移与灰度分发

**触发**：Python -> Rust/TS 的阶段性替换。  
**收益**：可回滚、低风险、可比较。  
**代价**：双栈维护期会拉高测试和运维成本。

### 不建议场景（反模式）

- 每请求都切 runtime（抖动与冷启动成本高）
- 没有冷却时间和熔断就自动切换（易震荡）
- 把业务策略硬编码进 spiderswitch（职责污染）

---

## 7. 建议落地路径

## Phase 0（立即）
- 明确 spiderswitch 边界：仅能力路由，不做策略决策
- 定义 runtime-neutral 能力模型和适配契约

## Phase 1（短期）
- 引入 runtime registry/resolver
- 支持 `runtime_id` 维度状态观测
- 保持上层策略下发，spiderswitch 仅执行

## Phase 2（中期）
- 增加 Rust 或 TS 运行时适配实现
- 建立跨运行时契约测试（同输入同语义输出）

## Phase 3（长期）
- 扩展 Go/WASM 适配器
- 建立统一基准：延迟、成功率、恢复时间、成本效率

---

## 8. 最终建议

1. **保持策略上移**：spiderswitch 只提供“可路由能力 + 可观测信号 + 执行接口”
2. **优先做抽象，不先做重写**：先统一 runtime 契约，再决定 Rust/TS 的实现优先级
3. **Rust 与 TS 并非二选一**：建议“Rust 核心控制 + TS 集成层 + Python 参考实现”并行演进
4. **提前为 Go/WASM 预留契约位**：避免未来再次架构重构
