# Eos 上下文架构 Phase 2+ — 动态分层结构化 & 外部文档化

> **版本**: v1.1
> **日期**: 2026-06-02（v1.0），2026-06-07（v1.1: 任务型上下文策略 + 二级检索模型，不限本地云端）
> **来源**: 先生（Alex）与 Spider 讨论
> **状态**: 远期架构规划（尚未分配给 Phase 2 task）

## 1. 核心动机

当前 Phase 2 已完成的能力：

| 组件 | 能力 | 限制 |
|------|------|------|
| `ContextBudget` | 按 token budget 倒序截断（linear flat + tool folding） | 无消息层级，一刀切 |
| `MessageAssembler` | 保证最小尾消息数，folding 大 tool 块 | 不区分消息重要性 |
| `SessionMirror` | 浏览器端会话镜像，ArtifactRef 大块引用 | 仅有存储抽象，无智能召回 |

先生（2026-06-02）指出的方向：

> **动态分层结构化**：按角色、权重、类型对消息分层，组装时按层级优先级裁剪，不是简单从尾巴往前砍。
>
> **外部文档化**：对话历史可以作为结构化文档持久化、可检索、可注入。新会话可以加载旧会话的决策摘要；长对话可以归档成外部文档，只带索引进当前上下文。

## 2. 设计原则

1. **分层 > 截断**：消息不只有"时间顺序"，还有"重要性层级"
2. **检索 > 保留**：尽量按需召回，不要盲目全量携带
3. **确定性 > 黑盒**：分层逻辑可控可测，完全避开 LLM compaction 风险
4. **渐进落地**：先从浏览器端索引做起，再扩展服务端
5. **任务感知 > 一刀切**：不同任务场景（编码/chat/review/debug/doc）按预制策略模板分配 Layer 0-5 注入优先级和 Budget
6. **最便宜的模型做向导**：选用最快或最便宜的模型代替传统向量索引做文件级检索，理解语义而非仅匹配关键词。不限本地还是云端，由成本和延迟决定

## 3. 消息分层模型

每条消息（或消息块）携带一个 `Priority` 标签，用于组装时的决策：

```
Layer 0 — System:    不可裁剪（system prompt, 工具定义, 行为约束）
Layer 1 — Active:    当前 round 刚发的消息（完整保留）
Layer 2 — Relevant:  被当前问题引用的历史消息或被检索命中的代码片段（带召回信号，完整保留）

**检索机制**：Layer 2 的填充可通过一个廉价或快速模型（如 Gemini Flash、GPT-4o mini、本地 1-3B 小模型）执行语义检索。该模型不参与主力推理，仅在上下文组装阶段作为"向导"——扫描文件树、定位相关函数/段落、输出相关文件+行号。主力 Provider 只消费检索到的精确片段，而非全量代码。

```
任务输入 → [快速检索模型] → 命中文件:行号 → [主力 Provider] → 输出
   (廉价)                       (仅相关片段)       (旗舰或无上限模型)
```
Layer 3 — Summary:   已归档的外部摘要文档（注入为 system，极简）
Layer 4 — Background:普通历史对话（按 budget 倒序保留，可选折叠）
Layer 5 — Archive:   已归档的长对话（提供索引/引用，不展开）
```

**组装策略**：从 Layer 0 → Layer 4 依次填充，直到触达 `ContextBudget` 上限。
Layer 5 不展开，仅提供 ArtifactRef 供搜索召回。

## 4. 外部文档化

### 4.1 文档存储

```
对话历史
 ├─ 活跃会话（浏览器 IndexedDB / OPFS）
 │   ├─ messages.json（元数据 + 头部/尾部消息）
 │   └─ artifacts/（大块 tool 输出、图片 base64、长代码）
 ├─ 外部文档（持久化存储，可选云端）
 │   ├─ {session_id}.summary.yaml（结构化摘要）
 │   ├─ {session_id}.full.jsonl（全量归档 JSONL）
 │   └─ {session_id}.artifacts/（大块引用）
 └─ 索引（关键词 → session_id + message_id）
```

### 4.2 摘要文档生成（未来）

当对话达到一定规模时，可选（用户触发或自动）生成对话摘要文档：

```yaml
# session_summary.yaml
session_id: "eos-xxxxx"
created_at: "2026-06-02T10:00:00Z"
provider: "deepseek"
model: "deepseek-chat"
message_count: 47
token_total: ~32000
topics:
  - "架构设计讨论"
  - "Prism 路由策略"
decisions:
  - topic: "路由插件化方案"
    consensus: "WASM first"
    ref: ["msg-23", "msg-25", "msg-28"]
attachments:
  - type: "code"
    path: "artifacts/prism-route-trait.rs"
    ref: "msg-31"
```

这个摘要文档本身可以注入为新会话的 system prompt 前缀（Layer 3），让新对话"知道谈过什么"。

### 4.3 外部文档召回（Phase 3+）

在新对话中，如果用户提问触发了关键词匹配，可以选择性地将历史对话的 **相关片段** 召回注入当前上下文：

```
用户: "上次我们讨论的路由策略方案，最终选了哪个？"
     ↓
查询索引 → 匹配 session=xxx 的 decision 条目
     ↓
召回包含 msg-23/25/28 的压缩消息块
     ↓
注入到当前请求 `messages` 的 Layer 2（Relevant）
```

## 5. Eos / Vela / Prism 中的职责

| 产品 | 职责 |
|------|------|
| **Eos 浏览器端** | IndexedDB 存储、SessionMirror 扩展（Priority 标注）、摘要生成触发 |
| **Vela 客户端** | 同上 + 可选 E2E 加密同步到云端（Phase 3） + **任务类型识别 + 策略模板加载 + 二级检索模型调用** |
| **Vela 本地侧端** | 二级检索模型推理（本地或选择最便宜的云端 API），执行代码文件检索，输出结构化检索结果供主力 Provider 消费 |
| **Prism** | 不存储对话内容。仅元数据（UsageRecord）涉及 token 统计。外部文档化完全是客户端侧业务 |
| **ai-lib-rust** | `MessageAssembler` 可选接受 Priority 标注输入（trait 扩展点），但核心不绑定分层模型 |

## 7. 不做的事情

- ❌ LLM 生成对话总结（用确定性摘要，避免质量波动和 compaction loop）
- ❌ Prism 存储对话内容（客户端责任）
- ❌ JavaScript 中的 Token 计数精确到字节（用 ai-lib-rust WASM 提供的 `estimate_tokens` 即可）

## 8. 验收准则（远期）

- 一次 200 轮的对话，`ContextBudget` 组装后 < 16K tokens，但关键决策点不丢失
- 新对话输入"上次那个方案选的是什么"，能召回历史相关片段
- 浏览器 IndexedDB 存储 100+ 长对话无性能退化

## 9. 上下文策略模板（v1.1 新增）

不同的任务场景需要不同的上下文组织方式。EOS 分层模型提供可配置的策略模板，每个模板定义 Layer 0-5 的注入优先级、检索目标和 Budget 分配。

### 9.1 任务类型识别

Vela 收到任务时首先识别类型（可通过指令提示词标注或模型自主分类）：

| 任务类型 | 标识信号 | 典型场景 |
|:---------|:--------|:---------|
| `chat` | 纯自然语言提问，无代码引用 | 问答、闲聊 |
| `code-fix` | 已知目标文件或错误输出 | 修 bug、改函数 |
| `code-review` | PR diff 或 commit | 审查变更 |
| `design` | 新功能/架构约束 | 设计模式、API 规划 |
| `debug` | 错误日志或异常堆栈 | 定位问题根因 |
| `doc` | API 接口或文档请求 | 生成/更新文档 |

### 9.2 策略模板矩阵

```
┌──────────────┬──────────┬──────────────┬──────────────┐
│ 任务类型      │ Layer 0  │ Layer 2 检索 │ Layer 3 摘要  │
├──────────────┼──────────┼──────────────┼──────────────┤
│ chat         │ 系统提示  │ 空           │ 历史摘要（轻）│
│              │ + 任务    │              │              │
├──────────────┼──────────┼──────────────┼──────────────┤
│ code-fix     │ 任务详情  │ 目标文件全量  │ 项目结构概览  │
│              │ + 已知路径│ + 相关函数    │              │
├──────────────┼──────────┼──────────────┼──────────────┤
│ code-review  │ PR diff  │ 被修改函数    │ 模块架构图    │
│              │ + 上下文  │ 的上下游引用  │              │
├──────────────┼──────────┼──────────────┼──────────────┤
│ design       │ 约束说明  │ 空           │ 全量架构图    │
│              │ + 目标描述│（不锁死代码） │ + 已有关联系统│
├──────────────┼──────────┼──────────────┼──────────────┤
│ debug        │ 错误信息  │ 相关日志     │ 模块依赖关系  │
│              │ + 环境    │ + 堆栈函数链  │              │
├──────────────┼──────────┼──────────────┼──────────────┤
│ doc          │ 接口规范  │ 该接口实现    │ 不相关文件    │
│              │ + 输出格式│ + 调用示例    │ 裁剪          │
└──────────────┴──────────┴──────────────┴──────────────┘
```

### 9.3 二级检索模型的角色

策略模板定义"要检索什么"，二级模型（廉价模型）执行实际的检索：

```
策略模板指示："检索目标文件 + 相关函数"
         ↓
[廉价模型] 读取文件列表 + 语义理解任务
         ↓
输出：相关文件路径 / 相关函数行号 / 段落索引
         ↓
主力 Provider 只接收检索到的精确片段（Layer 2）
```

二级模型的约束：
- 不需要理解复杂业务逻辑，只需判断"这段代码跟任务有关吗"
- 选用最便宜或最快的可用模型，不限本地还是远端（如 Gemini 1.5 Flash、Llama 3.2 1B、GPT-4o mini）
- 输出结构化（JSON 格式：target_files, related_functions, line_ranges）
- 出错时 fallback：模型可以继续通过 read 工具逐文件补充

### 9.4 二级模型的部署形态

| 形态 | 延迟 | 隐私 | 成本 | 适用场景 |
|:----|:----|:----|:----|:--------|
| 浏览器端 ONNX/WASM | ~10-50ms | ✅ 本地 | 免费 | chat、简单 doc |
| 本地 server（Vela 侧/VelaOS） | ~5-10ms | ✅ 本地 | 一次部署 | 编码场景全量，隐私敏感 |
| 云端 API（廉价 Provider） | ~200-500ms | ❌ 外发 | 按量付费 | 无本地算力时的兜底；选用 Gemini Flash / GPT-4o mini 等低成本模型 |

**部署优先级**：本地可用 → 用本地（零网络延迟，零按量付费）；本地不可用 → 选最快或最便宜的云端 API；两者可用 → 按策略（成本优先选本地，延迟敏感选本地，隐私敏感必须选本地）。

### 9.5 与现有 EOS 6 层模型的集成

策略模板不是替代 6 层模型，而是给每层定义了**按场景不同**的注入规则：

```
请求入口
  ↓
[1] 识别任务类型 → 加载对应的模板
  ↓
[2] 模板定义 Layer 0-5 的注入优先级和 Budget 分配
  ↓
[3] 二级检索模型按模板指示执行检索 → 填充 Layer 2
  ↓
[4] 从 Layer 0 → Layer 4 依次填充 ContextBudget
  ↓
[5] 提交给主力 Provider
```
