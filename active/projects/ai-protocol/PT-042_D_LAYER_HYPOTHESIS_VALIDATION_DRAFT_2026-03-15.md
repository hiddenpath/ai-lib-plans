# [D] 能力结构化演进假设-验证草稿（PT-042）

> 文档类型：Design Layer (D)  
> 目标：将“Capability 五元组（I/O/P/S/C）”从愿景陈述降阶为可验证、可回滚的执行提案。

## 0. 文档元信息

- 报告标题：`AI-Protocol Capability 五元组演进（假设-验证版）`
- 目标能力（如 I/O/P/S/C）：`Inputs / Outcomes / Process / Systems / Contract`
- 影响范围：`ai-protocol 文档治理、v2 manifest 设计约束、应用端能力索引策略`
- 版本：`draft-v0.1`
- 作者/评审人：`@hiddenpath / pending`

## 1. 问题定义

- 当前痛点：
  - 现有能力声明对复杂生成式任务表达不足（如异步视频生成、复杂输入约束）。
  - 报告中“事实”和“提案”混写导致可审计性下降，影响跨团队一致理解。
  - Runtime 与 Agent 的职责边界在文档层表述不够严格，存在“运行时变重”风险。
- 触发条件：
  - 供应商能力复杂度上升（多模态、长任务、工具生态）。
  - 业务方需要“按契约路由”而非“按模型名路由”。
- 非目标：
  - 本阶段不引入 runtime 行为改造。
  - 本阶段不将五元组字段强制入 required gate。
  - 本阶段不主张“行业标准化”结论。

## 2. 设计提案

- 提案概述：
  - 采用“文档先行、schema 后置”的 A 方案，将五元组先作为 D 层元数据提案治理。
  - 在不变更 runtime 核心执行路径的前提下，先验证“声明式能力描述”是否提升路由与审计质量。
- 核心结构：
  - `I`：输入约束（模态、数量、大小、采样率等）
  - `O`：输出语义（如 1080p 视频、结构化推理产物）
  - `P`：过程形态（同步/异步轮询、状态机）
  - `S`：外部系统依赖（MCP/GIS/Search 等）
  - `C`：契约边界（错误模型、安全规则、成本约束）
- 与现有协议兼容性：
  - 初始阶段仅为可选元数据，不破坏当前 v1/v2 manifest 解析路径。
  - 通过 report-only 证据门禁评估成熟度，再决定 schema 正式化节奏。
- 与 runtime 的边界：
  - Runtime 仅透传/暴露 capability profile，不承担复杂推理执行。
  - 复杂决策保持在应用层（Router/Orchestrator/Policy）完成。

## 3. 假设与验证

| 假设ID | 假设内容 | 验证方式 | 通过标准 | 失败处理 |
|---|---|---|---|---|
| D-H01 | 五元组可提升复杂能力表达完整度 | 对比现有 boolean capability 与五元组字段覆盖率 | 试点模型关键能力覆盖率 >= 90% | 退回 I/O/S 子集，暂缓 P/C |
| D-H02 | Runtime 保持轻量仍可支持能力发现 | 仅新增 profile 暴露，不改执行链；评估改动范围 | runtime 无新增核心执行分支 | 明确禁止 runtime 解释 P/C |
| D-H03 | Agent 可利用 I/O/S 改善路由确定性 | 在试点任务中比较命中率/回退率 | 命中率提升且回退率下降 | 收窄到单域能力标签 |
| D-H04 | P 描述可减少异步任务失败重试抖动 | 对异步模型任务流程做状态机对照 | 异步超时/误重试下降 | P 仅保留同步/异步二元标记 |
| D-H05 | C 可增强治理审计 | 用证据矩阵对 C.security / C.cost 做审计抽查 | 关键契约项可追溯率 100% | C 改为文档约束，不入 manifest |
| D-H06 | report-only gate 可控且不影响发布节奏 | 跑一轮 report-evidence-gate 试运行 | 无发布阻塞，且问题可收敛 | gate 继续保留人工审查模式 |

## 4. 实施计划

- P0（最小试点）：
  - 完成 D 层文档改写，显式标注假设与非目标。
  - 在 DeepSeek（推理）与 Seedance（视频异步）建立五元组样例描述（文档层）。
  - 输出 Claim-Evidence Matrix（含 E1-E4 分级）。
- P1（扩展验证）：
  - 扩展至 2-3 个额外模型类别（如图像生成、工具密集型）。
  - 对比“按模型路由”与“按能力签名路由”的质量指标。
- P2（流程化推广）：
  - 固化 report-evidence-gate（先 report-only）。
  - 形成周度复核 + 双周演进评审节奏，待稳定后再讨论 required 化。

## 5. 风险与回滚

- 技术风险：
  - 字段语义过宽，导致跨团队理解不一致。
  - P/C 过早进入结构化约束，增加实现复杂度。
- 兼容性风险：
  - 若被误解为“已实现能力”，会造成对 runtime 预期偏差。
- 回滚触发条件：
  - 试点阶段出现显著语义歧义、审计不可复核、发布流程被拖慢。
- 回滚步骤：
  1. 停止扩大五元组覆盖范围；
  2. 回退为 I/O/S 子集；
  3. P/C 保留在 D 层文档，不进入结构化强约束。
