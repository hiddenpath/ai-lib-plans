# ZeroSpider ai-lib Migration — Phase Completion Audit

**Date**: 2026-04-27
**Auditor**: Spider (review side)
**Scope**: `ZEROSPIDER_AI_LIB_MIGRATION_PLAN.md` Phase 0–6 execution checklist vs actual codebase state (`main @ aa3214a`)
**Verdict**: **6 tasks closed, but 5 of 7 phases have material gaps. Plan `status` should NOT be `completed`.**

---

## Executive Summary

ZS-ML-001~006 are all marked `completed` with PR links and merge commits backfilled correctly. However, the task YAMLs represent a **narrow interpretation** of what each Phase required — they delivered documentation, config examples, feature gates, and error hints, but **left the core runtime logic largely untouched**. The migration plan's execution checklists describe work that was not done, and the plan status was set to `completed` prematurely.

**Key finding**: the migration delivered the *scaffolding* (docs, feature flags, CI gates, config examples) but not the *substance* (dependency alignment, adapter API parity, fallback runtime, legacy deletion, security audit). ZeroSpider today still resolves providers through the same `create_provider_with_url_and_options` → legacy match arm path for all non-`provider/model` strings, and `ProtocolBackedProvider` is only reachable when the user explicitly writes a slash-separated id.

## Cross-Phase Gaps

### 1. Streaming Tool Call Handling — CRITICAL

`ProtocolBackedProvider::stream_chat_with_system` (line 336–380) handles `PartialContentDelta` and `ThinkingDelta` but **does not handle `PartialToolCall`**. Any agent using tools in streaming mode will silently lose tool call data. This is a Phase 1 gap but affects all downstream phases.

### 2. Protocol-Aware Fallback — HIGH

`ReliableProvider` (reliable.rs) implements model_fallbacks for legacy providers. When `ProtocolBackedProvider` fails (e.g., manifest misconfigured, provider down), there is **no fallback chain**. The user's `[reliability]` config is ignored for protocol providers. Phase 4 gap.

### 3. `--no-default-features --features ai-protocol` Build — MEDIUM

The plan says Phase 5 should verify this build. CI never tests it. If any default feature leaks into the protocol path, it won't be caught until a user tries a minimal build.

### 4. Embeddings / Batch Feature Wiring — MEDIUM

`ai-lib-rust` is declared with `features = ["embeddings", "batch", "telemetry"]` but none of these are wired to any code path. This is dead dependency weight. Either wire them or remove the feature flags.

### 5. Protocol Registry CLI Surface — MEDIUM

`protocol_registry.rs` can scan manifests and list providers/models, but this capability is **not exposed** through any CLI command. The existing `zerospider models` command only lists legacy providers.

## Recommendations

### R1. Reopen Plan Status

`ZEROSPIDER_AI_LIB_MIGRATION_PLAN.md` status should be **`in_progress`**, not `completed`. At minimum, Phases 1, 4, and 5 require new task YAMLs.

### R2. Create Missing Task: ZS-ML-007 (Phase 1 — Adapter API Parity)

**Scope**:
- Add `PartialToolCall` handling to `ProtocolBackedProvider::stream_chat_with_system`
- Add multi-turn tool call conversion test (3+ messages with tool role)
- Add model override unit test
- Document API alignment with ai-lib-rust 0.9 recommended patterns
- Add streaming test with tool call events

**Acceptance criteria**: streaming tool calls work end-to-end; test coverage ≥ 5 cases for `convert_messages` + streaming.

### R3. Create Missing Task: ZS-ML-008 (Phase 2 — Registry CLI + Availability)

**Scope**:
- Wire `protocol_registry` scan results to `zerospider models protocol-providers` / `protocol-models` CLI
- Implement env-availability filter: only show providers whose required env vars are all set
- Add CLI integration test

**Acceptance criteria**: `zerospider models protocol-providers` lists actual manifest providers; providers with missing keys show as unavailable.

### R4. Create Missing Task: ZS-ML-009 (Phase 3 — AiClient as Default Path)

**Scope**:
- When `default_provider` matches a manifest provider_id (even without `/`), route to `ProtocolBackedProvider`
- Make `resolve_ai_client` the actual runtime entry, not just a helper
- Integration test: user sets `default_provider = "openai"` (no slash) → resolves via manifest if available

**Acceptance criteria**: non-slash provider names resolve through protocol when manifests exist; legacy path only used as fallback when no manifest matches.

### R5. Create Missing Task: ZS-ML-010 (Phase 4 — Protocol Fallback + Metrics)

**Scope**:
- Implement `ReliableProvider` wrapper for `ProtocolBackedProvider` with `model_fallbacks`
- Wire `AiClient::metrics()` to existing `/metrics` endpoint (if ai-lib-rust exposes it), or document explicit defer with rationale
- Add resilience boundary test: prove no double-retry between `execute_chat_with_retry` and `[reliability]`
- Audit OTel metric overlap

**Acceptance criteria**: protocol provider failure triggers configured fallback chain; no double-retry under any path.

### R6. Create Missing Task: ZS-ML-011 (Phase 5 — Legacy Code Removal)

**Scope**:
- Inventory all `providers/*.rs` logic that has no manifest equivalent
- For items with manifest coverage: delete code under `legacy-providers` gate

- Major version bump + CHANGELOG breaking section

**Acceptance criteria**: `--no-default-features --features ai-protocol` build + test passes; legacy code volume reduced by ≥ 50%.

### R7. Update ZS-ML-006 Completion Notes

ZS-ML-006 should note that embeddings/batch wiring and systematic security audit are **deferred**, not completed. Add a `deferred_items` section to the YAML.

### R8. Clean Up Dead Feature Flags

Either wire `embeddings`, `batch`, `telemetry` ai-lib-rust features to actual code paths, or remove them from `Cargo.toml` to reduce dependency bloat and build time.

### R9. One Task → One PR (Already Agreed)

Future tasks must follow one-task-one-PR discipline. The ZS-ML-003~006 batch PR caused review complexity and made gap tracking harder.
